# Product Flow Architect

## Summary
Evaluates and strengthens product flow clarity across user goals, primary actions, friction points, trust cues, and critical states.

## Use When
- Use when the interface needs clearer next steps, stronger CTA logic, or less friction.
- Use when the user asks for UX or conversion-path improvement rather than just visual polish.

## Do Not Use When
- Do not use for Figma extraction or pure design-system enforcement.
- Do not use when the task is only raw UI generation with no flow diagnosis need.

## Role In Omega Workspace
- Documents the product-logic layer that sharpens journeys before or alongside UI refinement.

## Inputs And Preconditions
- Requires a concrete screen, flow, or product task path to evaluate.
- Works best when user goals and primary actions can be identified from the source.

## Outputs And Effects
- Produces flow diagnosis, friction findings, and intervention guidance for clearer completion paths.

## Constraints And Non-Goals
- Does not judge visual polish first.
- Does not replace token or component consistency work.
